What changes, and what it costs

Green = rib-only, ships from this repo. Amber = needs a keelson host change.

The briefing: the claim wears a card, the receipt wears a line rib-only boards/cast.ts · briefSection()

The four boxed panels had a real bug under the sameness: they dressed a model's claim and a harness's count in identical chrome. the thesis is prose a confabulation can produce; the scan is a number counted off the turn's tool_use chunks. Same box, same label column, same weight — the board was flattening the one distinction it exists to draw.

So the form now encodes the voice. The claim gets a card — the same anatomy as the seats it's proposing, so the board reads as one grammar: the squad's card, then its members' cards. Title is the ensemble, the pill is capacity (absorbing the old 5 of 6 bench tile), and the thesis sits under the card's own divider where a seat's purpose sits. The provenance gets bare lines — no brief given, and what the scan actually read, still expanding to the file list. A card is an assertion; a line is a fact. You can tell them apart at a glance now, which four identical panels never let you do.

It's also about 40% shorter, and the · chain (26 files read · 7 searches (glob / grep)) says in one line what three panels said in three.

One thing I tried and backed out

Putting the no-brief warning inside the card as a toned field. It's the only toned text slot a card has — reason and footnote take no tone — but field values render in monospace, so a prose warning came out looking like code. Hence the warning is a row: rows aren't mono, and the glyph carries the tone. Worth knowing if you reach for a card field for prose later — it will fight you.

The card lost its labels — both of them rib-only boards/cast.ts · cardFor()

Both were free: reason.label and a field's label are each optional in the schema. So purpose: drops to bare description, and can drops to bare capability.

The divider you wanted is already there and always was — .cvb-card-reason carries a 1px dashed top border. Removing the two labels is what finally lets it read as a divider rather than decoration: capability above the rule, purpose below it. Two zones, no words spent naming them. The briefing card reuses the same rule under its title.

Capability: the mark is the noun rib-only boards/cast.ts · cardFor()

With can gone, the line has to say "capability" by itself, so the encoding carries it: ✎ code, read in caution amber for a seat that can modify your repo — read in dim, unmarked, for one that can't. The asymmetry is deliberate. Read is the baseline every seat has; write is the exception worth a mark, so only the exception gets one. Scan the bench and the amber pencils are the answer to "who can touch my code" without a legend, a pill, or the word "can".

Two channels, on purpose: app.css's own identity rule is that colour never carries a signal alone ("the always-present name is the relief channel"). The ✎ and the word code both survive a viewer who can't see amber.

The flip is real — but it costs a file on disk rib-only new: cast-view.json · bin/collect-cast.ts

Turning the card over works, and read-only makes it simpler than the editable version: no form, no submit, no cast-edit-charter. The card keeps its head — dot, name, role pill — so the identity holds across the turn, and the body swaps to the charter as stacked fields, a primitive the host built for exactly this ("one field per line ... where the break IS the shape"). The charter's own sections are the field list: Personality, Backstory, Role, Mission, Voice. One ↩ back.

The catch is where "which card is flipped" lives. squad-cast is a bash collector — bin/collect-cast.ts runs out-of-process — so module state in index.ts is invisible to it. The flip has to be a marker file the collector reads. That's not a hack; it's the roster's existing pattern: collect-roster.ts already reads pending-genesis.json to decide whether to draw a boot card. So: cast-view.json holding { flipped: slug }, written by a cast-flip verb, read beside the proposal. Same round-trip cost as cast-pick, which already writes disk and respawns the collector every time you drop a seat.

The card no longer resizes — it scrolls, as you asked. Worth knowing why that needed a specific fix: my first attempt put overflow:hidden on the card, which is the usual trick, and it did nothing. An auto grid row sizes to its items' max-content; overflow only zeroes an item's automatic minimum. The cap has to be an explicit max-height on the scroller. That's the version in the mock, and Wash's dead gap drops from ~350px to ~40px.

Which makes the number a real decision, not a detail: 150px is a magic constant, tuned so a flipped card lands near a front card's height. It buys a stable bench and costs a reading window — ~8 lines of a ~22-line charter, so Simon's takes about three scrolls. Raise it and the bench breathes unevenly again; lower it and you're reading through a letterbox. If that trade reads badly in use, the escape hatch is already in this rib: view-run returns { effect: "open-canvas", key, title }, and a view-charter could do the same — full width, no cap, no distortion, at the cost of leaving the bench.

Three small things the contract won't give you — one host PR host change keelson · .cvb-card-actions / .cvb-card-fields--stacked / canvasActionItemSchema

Everything else on this board ships from the rib. These three are layout and layout belongs to the host — the board declares what, the host decides where. They're all small, all scoped, and they land together:

1 · The corner. .cvb-card-actions is a plain left-aligned flex-wrap row and a producer can't set justify-content. The host already does this trick one element up — .cvb-card-overflow { margin-left: auto } pins the ⋯ to the head's right edge. So: an action hint (align: "end", additive like inline and subtitle), or a rule right-aligning a lone trailing icon. Blanket justify-content: space-between would be the wrong fix — it would push the roster card's Retire away from its Enter too.

2 · The scroll cap. .cvb-cards--grid .cvb-card-fields--stacked { max-height; overflow-y: auto } — nothing in app.css caps a card's height today, so this is genuinely new. Scoped to grid + stacked, which is why it can't disturb the run-loop's stacked readouts.

3 · Icon-only. No iconOnly flag exists (labelContent = item.label, always rendered), so ▤ today means label: "▤" — which makes the button's accessible name the glyph rather than "Charter". hint already gives the tooltip for free.

It's a small, well-precedented add. The host already does exactly this trick one element up: .cvb-card-overflow { margin-left: auto } pins the ⋯ menu to the head's right edge. So either an action hint (align: "end" → margin-left:auto, additive like inline and subtitle before it), or the host right-aligning a lone trailing icon action by rule. Blanket justify-content: space-between would be the wrong fix — it would push the roster card's Retire away from its Enter too.

Bundle it with the other icon-button gap: there's no iconOnly flag either (labelContent = item.label, always rendered), so ▤ today means label: "▤" — which makes the button's accessible name the glyph. hint already gives you the "Charter" tooltip for free. One small host PR fixes placement and the screen-reader name together.

Still open decide

Read-only settles provenance. Dropping charter editing removes the question I raised last round — every charter on the board is the scan's, so the receipt's evidence still describes what you're approving, and no "edited by you" footnote is needed. Worth noting the charter is already mutable after approve, on the member, via the roster. This board is now purely a decision surface: read, drop, retune the model, approve.

The rationale is still homeless. Purpose took the card's one prose slot, so the scan's per-seat argument is persisted but unrendered. The collapsed the scan row answers "was this grounded?"; nothing answers "why this seat?". The card back has room — the flip could carry why cast as a sixth stacked field, under Voice. That would make the back the full story of the seat: who it is, and why the scan wanted it.
