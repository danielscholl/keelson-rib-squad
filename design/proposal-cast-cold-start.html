Cold start: the cast screen, recomposed

The screen has one job — get a first-time operator from an empty scope to a squad — and it has the priority right: Cast is the hero, seating one member is the escape hatch. What's wrong is the shape. The escape hatch renders as five full-width tone-washed bars that out-mass the hero four to one, the hero is a button hiding the only input that matters, the footer restates the header, and the screen never says which repository it is about to read.

Six changes, all rib-only. Every field used here ships today — no @keelson/shared change, so nothing waits on a contract landing in keelson main. Static mock — not the live app.

Status — shipped, with two corrections found building it

Landed: the five rib-side changes, green on bun run check, bun run typecheck, and 992 tests. Two things this mock originally claimed were wrong; the code is right rather than the drawing. Both are corrected above, and recorded here because a mock that quietly diverges from what shipped is worse than no mock.

1 · hint is hover-only. The first draft drew "Reads ~/keelson/cimpl-stack · creates nothing" as a visible line beside the submit. It is not expressible: hint feeds actionTooltip (BoardView.tsx:118) and renders as a title tooltip, invisible at rest. The path echo moved to the intro row's trailing — "a short right-aligned annotation" — which is visible and needs no new primitive. hint remains exactly right for the archetype taglines, where a hover reminder is the intent.

2 · The name must resolve off projectId, never scopeId. DEFAULT_SCOPE_ID and DEFAULT_PROJECT_NAME are the same literal — "default" — with different meanings, so a scopeId-keyed lookup could match a project whose id is "default" and chip the sentinel as though it were a repo identity. Resolution is snapshot-first keyed on projectId, with the frozen selection.name as fallback: projects.json is rewritten on every action while selection.name froze at select time, so after a rename the snapshot is right and the selection is stale.

What the trace turned up

The launchpad's five identity hues are a promise the write path breaks on the first click. authorSection tones each archetype by its position in the array — tone: identityToneForSlot(i) — so Lead wears id-blue, Engineer id-amber, Reviewer id-teal, Tester id-rose. But nothing carries that slot into the write. authorArchetypeAction sends { role } and a brief — no slot — and squad_emit_member assigns the hue from cast order: themedRecord(…, existing.length) → identitySlotForIndex(0) on an empty roster.

So click the rose Tester on a fresh scope and the boot card that replaces it is blue (bootCard(pending, nextFreeSlot([])) → slot 0), and the member it becomes is blue for life — identitySlot is persisted, and the dot, the roster peek, and every run board key off it. On an empty roster every archetype authors into id-blue: four of the five chips are wrong, immediately and permanently.

The ramp is right; the preview is wrong. Slot-by-cast-order is deliberate — it's what stops a hand-built team stacking on slot 0, and reserving rose for Tester would give an all-Tester roster five rose seats. So the fix is not to teach the write path the archetype's hue. It is that the launchpad cannot know the hue, and must stop claiming one. Which is convenient: those tones are also the full-width gradient washes that make the escape hatch shout over the hero. One deletion, two problems.

Direction, scope, and trade-off calls

The escape hatch went from ~250px of stacked column to one ~34px line. The taglines aren't lost — they move to hint, the schema's "short descriptive hover tooltip… so a producer can remind the operator what an unfamiliar action is for" (shown hanging off Lead). The hue washes are gone, so the only saturated thing left is the verb the screen exists for. Nothing here is a new primitive.

Detail · Describe & author, folded in Why the fifth chip doesn't hang: in a wrap strip a fields-carrying action is a compact chip at rest and only claims its own full-width line once opened. It is one chip in the row above, and the panel below is the same action, clicked.

Who should this member be?
The mechanic — and the one constraint it carries

The rule is .cvb-actions--wrap > .cvb-action:has(.cvb-action-form) { flex-basis: 100%; }. The selector is :has(), and .cvb-action-form only exists in the DOM while the form is open — so at rest the item is flex: 0 0 auto like every sibling and sits in the line. The CSS says so in its own comment: "…breaks to its own full-width line so the form still reads as a panel beneath the row (its button stays compact, the form stretches)." This is the behaviour hireSection already relies on.

The constraint: Describe must stay last in items. A wrap strip breaks the open form in source order, not below the strip — that "never breaks mid-row" guarantee belongs to tabs, which sends the form below via order. So Describe last means the four archetypes hold line one and the form drops cleanly to line two. Move it anywhere else and opening it splits the chip row in half. It is last today; this is a note for whoever reorders the array next.

Making the project impossible to miss

The failure this prevents is expensive and silent: land on Squad, click the one big button, and a 300-second read runs against whichever repository the chip happened to hold — proposing a team into the wrong scope. On an empty roster there is no seated member to tip you off that you're in the wrong place. The chip is the only tell, and it is a small quiet pill in the opposite corner from the button.

What the rib can and cannot do

The picker is not the rib's to style. Surface.tsx renders it from descriptor.projectScoped as <ProjectChip projectName={…} popoverId={…} /> — three props, no prominence among them — and ProjectChip is the shared chat-model-chip, the same component and the same class Chat and Workflows use for their model pill. So there is no rib-side change that makes the control louder, and restyling that class to shout would drag Chat and Workflows along with it. A prominence variant for projectScoped surfaces is a real option, but it is a keelson PR — and CI here symlinks @keelson/shared to keelson main, so this rib can't land ahead of it.

So make the answer dominant, not the control. What the operator needs isn't a bigger picker — it's to never be able to reach the button without having read the target. Three echoes on the way there, all rib-side, all shipping fields, and no second control to keep in sync:

1 · The panel head states its scoperib · roster.ts

header.chip is free at cold start: it carries the hoisted ensemble on a seated roster, and hoistedEnsemble([]) returns undefined for an empty one — there is no ensemble to collide with until there are members, and by then the cold-start board is gone. So the head reads Roster · 0 members · cimpl-stack, in the same pill that will later say THE FLUX NAVIGATORS. Top-left, where the eye starts, at zero cost.

2 · The verb names its targetrib · roster.ts · collect-roster.ts

Cast a squad from cimpl-stack, not "from this repo". The collector already reads the selection (readSelectedProject, for selectedScopeId) and liveRunsStrip already renders run.name ?? run.scopeId — the name is in hand and just needs threading into buildRosterBoard. Degrades to today's "this repo" on the DEFAULT_SCOPE_ID sentinel, which has no name to give.

3 · The framing line says whererib · roster.ts

The invitation row already carries the trust claim — the scan reads, it does not write — so the root rides its trailing: "One scan of the repo composes the team — you approve before anything is created." with ~/keelson/cimpl-stack right-aligned beside it. That is the last prose before the verb. The name says which; the path says where, and disambiguates two checkouts sharing a name. rootPath is already on the selection record — and has no fallback (projects.json carries only { id, name }), so it degrades to silence rather than a guess.

If that isn't enough, the rib can ship a real contextual picker without touching the harness: an actions + wrap strip of select-project chips, one per project. The precedent already ships — liveRunsStrip dispatches exactly that verb with payload: { scopeId } — and it can't drift from the host chip, because both write the same file through the same handler. It is left out of the recommendation only because it is a second control for one piece of state, and because three echoes solve the actual failure without one. Worth adding the moment we see someone cast against the wrong repo anyway.

Considered and rejected

Each of these is a primitive that looks made for this screen and isn't. Written down so the next pass doesn't re-derive them.

An empty bench of ghost seatscanvas · cards.ghost / seats

The tempting one: make cold start look like a squad — a grid bench of dashed ghost seats, or the unused seats section ("a fixed-capacity identity row… unlabelled seats stay decorative so regions do not invent identities"). It's the shape Chamber's presence board uses, and it would make the empty state continuous with the seated roster. But it has to commit to a number, and there isn't an honest one. Five is the hue ramp (IDENTITY_SLOT_COUNT); six is MAX_CAST_MEMBERS; the cast itself proposes a variable team. Any fixed bench reports an internal cap the operator never chose — the exact ground the last pass deleted the 5 of 6 bench stat tile on. Chamber can draw five seats because authoring one Mind at a time is its interaction. Squad's is one scan that decides the number for you.

tabs for the archetype stripcanvas · actions.tabs

A tab strip of Lead / Engineer / Reviewer / Tester / Describe reads beautifully and lies. The four archetypes carry no fields — they dispatch a fixed brief on click — and in a tabs strip "items without fields still dispatch on click". So four of the five tabs would fire an action instead of revealing a panel. A tab that fires is a lie about the affordance, and the one tab that behaves like a tab would be the escape hatch's escape hatch. wrap is the honest primitive for a strip of immediate verbs — and it folds Describe in for free.

A dominant project picker on the boardcanvas · cards.action + selected

The literal reading of "make it dominant": a selectable cards bench of projects (action + selected gives a real ringed picker), sat above the hero. It reintroduces the bug this pass exists to fix. A grid of project cards out-masses the composer exactly the way five tone-washed bars do today — and it puts a control the operator touches once per session above the verb they came for. The dominant thing should be the answer ("you are about to read cimpl-stack"), not the control that sets it.

Keeping the Cast / Meet / Run explainerrib · roster.ts

It goes, and the duplication is only half the reason. Its Cast tile — "The scan proposes a team; you approve or discard it" — is the invitation row again, thirty words later. But Meet and Run describe panels that are hideWhenEmpty, so at cold start they explain two things the operator cannot see or act on, and the tiles re-render on every empty scope forever, not just the first one ever. The surface subtitle already says "Author members · cast a squad · assign work". If the arc needs telling, the honest place is docs, not a permanent footer on the create screen.
